# 安心办 MVP 开发计划草案

## 1. 目标

本开发计划的目标，是在不扩展非 MVP 功能的前提下，按最小可行路径完成“安心办”第一版 MVP，并确保在初赛演示中能够稳定展示以下主链路：

选择场景 → 粘贴说明 → AI 生成通俗解释 / 步骤卡 / 安全提醒 → 前端展示

本计划遵循以下原则：

- 先把主流程跑通，再做界面优化。
- 先保证老人能用，再考虑展示效果。
- 先做可演示版本，再做部署整理。
- 所有实现都必须遵守 PRD、UI、TECH 三份文档的边界。

## 1.1 初赛评分硬约束

后续开发、验收和演示准备，必须同时对齐初赛专业评审的四个评分维度：

- 创新性 30%：需求创新、解决思路、技术创新。
- 实用性 30%：场景洞察、解决效果、落地可行性。
- 完成度 20%：核心链路、用户体验、价值传递。
- 美观度 / 设计体验 20%：视觉设计、交互友好度。

这意味着后续每个阶段不仅要“做出来”，还要能明确回答：

- 我们的价值是否足够新。
- 我们的方案是否真的对老人有用。
- 我们的主链路是否足够完整。
- 我们的界面和交互是否足够友好、适合演示。

## 2. 范围

### 2.1 本期范围

本期只做以下 MVP 范围：

- 首页展示医保、银行、政务三个平权场景入口。
- 用户可以进入说明输入页并粘贴公开说明文本。
- 前端把场景和文本提交给后端。
- 后端调用 AI，生成统一响应外壳 `success + data + error`。
- `data` 内返回固定结构：通俗解释、步骤卡、安全提醒、风险等级、风险标签。
- 前端按 UI 文档展示结果。
- 高风险内容采用“双层机制”：AI 识别为主，后端关键词兜底。
- 高风险时前端置顶展示加强提醒，并显示“先找家人确认”“联系官方渠道”两个提醒动作按钮。
- 部署采用“前端静态页 + 一个轻量后端服务”的演示方案。

### 2.2 本期技术路线

按 TECH 文档执行：

- 前端静态页
- 轻量后端服务
- 统一响应外壳 `success + data + error`
- AI 识别 + 后端关键词兜底

## 3. 不做什么

本计划明确不包含以下内容：

- 不做登录注册。
- 不做历史记录。
- 不做 OCR 拍照识别。
- 不做语音输入。
- 不做真实医保、银行、政务接口接入。
- 不做真实业务跳转。
- 不做支付、验证码处理、电话呼起等真实操作。
- 不收集身份证号、银行卡号、密码、验证码、支付信息、真实医保账号等敏感信息。
- 不在 MVP 阶段扩展复杂后台、账户体系或统计系统。

## 4. 开发阶段拆分

本期开发按“先最小可跑通，再优化 UI，再部署”的顺序推进。

### 阶段一：主流程最小可跑通

#### 阶段目标

先把最核心的业务链路打通，确保用户能够从场景选择走到 AI 结果展示。

#### 主要任务

- 搭建前端最小页面结构：场景选择页、说明输入页、结果页、错误状态页。
- 实现前端场景选择与文本提交流程。
- 搭建轻量后端服务。
- 打通前端到后端的请求链路。
- 打通后端到 AI 的调用链路。
- 实现统一响应外壳 `success + data + error`。
- 产出固定结构化结果：通俗解释、步骤卡、安全提醒、风险等级、风险标签。
- 在前端展示通俗解释、步骤卡和默认安全提醒。

#### 阶段交付物

- 可访问的前端基础页面。
- 可响应请求的轻量后端服务。
- 可返回结构化结果的 AI 调用流程。
- 首条完整主流程演示版本。

#### 阶段验收标准

- 用户可以选择医保、银行、政务场景。
- 用户可以粘贴公开说明文本并提交。
- 后端可以成功返回 `success + data + error` 格式。
- `data` 中包含通俗解释、步骤卡、安全提醒。
- 前端可以正确展示结果页。
- 失败时前端可以展示明确错误提示。

#### 阶段一评分说明

- 已满足完成度中的“核心链路骨架已跑通”。
- 已初步满足实用性中的“主流程具备可验证形态”。
- 尚未满足创新性和实用性中的“真实 AI 能力证明”，因为当前 `callAi()` 仍是 fallback 骨架验证。
- 当前也未完成美观度 / 设计体验目标，因为老人友好 UI 仍未进入阶段二优化。

### 阶段二：高风险处理与 UI 优化

#### 阶段目标

在主流程跑通的基础上，补齐高风险处理和老人友好界面细节，让产品更可用、更适合比赛截图展示。

#### 主要任务

- 实现 AI 风险识别字段的消费。
- 接入真实 AI 调用，替换当前 fallback 骨架结果。
- 实现后端关键词兜底风险判断。
- 合并 AI 风险判断和关键词判断结果。
- 在前端高风险场景下置顶展示加强版安全提醒。
- 展示两个提醒动作按钮：“先找家人确认”“联系官方渠道”。
- 优化结果页信息顺序：普通场景先通俗解释，再步骤卡；高风险场景先安全提醒。
- 优化步骤卡的一屏一步浏览体验。
- 优化移动端显示、大字、高对比、少按钮体验。

#### 阶段交付物

- 高风险处理完整版本。
- 老人友好的结果页版本。
- 可截图、可演示的 UI 展示版本。

#### 阶段验收标准

- 高风险关键词可以触发兜底提醒。
- 高风险场景下安全提醒可以优先置顶。
- 两个提醒动作按钮只做引导，不触发真实业务。
- 普通场景首屏先展示通俗解释。
- 步骤卡支持上一步 / 下一步的一屏一步体验。
- 页面在移动端可读，老人可理解主要操作。

#### 阶段二评分重点

- 优先补足创新性：让真实 AI 能力真正参与生成，而不是只靠静态 fallback。
- 优先补足实用性：让高风险识别和通俗解释对真实输入更有效。
- 优先补足完成度：让主链路不只是跑通，而是更接近可交付体验。
- 优先补足美观度 / 设计体验：完成老人友好 UI 的关键优化。

### 阶段三：部署准备与演示收口

#### 阶段目标

把已经跑通并优化过的 MVP 整理为可演示、可访问、可截图保存的版本。

#### 主要任务

- 确认前端静态页可独立部署。
- 确认轻量后端服务可稳定提供 API。
- 确认 AI 密钥只保留在后端。
- 确认演示链路稳定可重复。
- 准备医保主案例作为首发演示样例。
- 准备错误场景和高风险场景示例，便于比赛展示。

#### 阶段交付物

- 可访问的演示地址或演示环境。
- 可重复演示的主案例。
- 可截图的首页、输入页、结果页、高风险页。

#### 阶段验收标准

- 前端静态页可正常访问。
- 后端服务可稳定响应。
- 主流程从输入到展示可完整走通。
- 医保主案例演示稳定。
- 高风险案例演示稳定。
- 页面适合截图与录屏展示。

#### 阶段三评分重点

- 把创新点、实用价值、完成度和设计体验统一收口到最终演示材料里。
- 确保评委在一次演示中能同时看到：真实 AI 价值、老人友好体验、安全提醒能力和可落地性。

## 7. 测试清单

每完成一个阶段，都要做本地验证，并按项目规则输出测试结果和剩余风险。

### 7.1 主流程测试

- 能否选择医保场景。
- 能否选择银行场景。
- 能否选择政务场景。
- 能否粘贴公开说明文本。
- 能否成功返回 AI 结果。
- 能否正常展示通俗解释。
- 能否逐步查看步骤卡。
- 能否始终看到安全提醒。

### 7.2 高风险测试

- 输入含“验证码”内容时，是否触发加强提醒。
- 输入含“转账”内容时，是否触发加强提醒。
- 输入含“收费”内容时，是否触发加强提醒。
- 输入含“陌生电话”或“链接”内容时，是否触发加强提醒。
- AI 未标高风险但关键词命中时，后端兜底是否生效。
- 两个提醒动作按钮是否只做提示，不触发真实跳转。

### 7.3 错误处理测试

- 未选择场景时是否提示明确。
- 输入为空时是否提示明确。
- 输入内容太短时是否提示明确。
- AI 调用失败时是否返回统一错误结构。
- 前端收到失败结果时是否展示用户可理解提示。

### 7.4 移动端与老人友好测试

- 首屏主要信息是否清楚可读。
- 字体是否足够大。
- 对比度是否足够明显。
- 按钮是否容易点击。
- 页面是否避免多按钮干扰。
- 高风险提醒是否足够显眼。

## 8. 风险与回滚策略

### 8.1 主要风险

- AI 输出结构不稳定，导致前端无法直接渲染。
- 当前 `callAi()` 尚未接入真实 AI，只能返回 fallback 结构化结果。
- AI 漏标高风险内容。
- 前端结果页信息过多，老人看不懂。
- 后端错误处理不统一，导致前端异常状态混乱。
- 演示环境不稳定，影响比赛展示。

### 8.2 应对策略

- 使用统一响应外壳，减少前端分支复杂度。
- 使用固定 JSON 结构，减少 AI 自由发挥空间。
- 在进入最终演示与部署前，必须完成真实 AI 接入。
- 接入真实 AI 后，失败时必须返回明确错误提示，不能继续用 fallback 假装成功。
- 用后端关键词做高风险兜底，降低漏标概率。
- 按“一屏一步、少按钮、大字高对比”持续收敛 UI。
- 先打通最小链路，再做 UI 优化，避免早期返工。

## 8.4 当前最大短板

当前最大短板不是页面骨架，而是真实 AI 能力尚未接入。

具体来说：

- 当前 `callAi()` 还只是 fallback 骨架验证。
- 这意味着我们现在能证明“链路存在”，但还不能证明“AI 价值已经真实成立”。
- 这会直接影响创新性、实用性和完成度三个评分维度。

因此，在后续开发优先级上，真实 AI 接入必须排在高位。

### 8.3 回滚策略

- 如果阶段二的复杂提醒逻辑不稳定，先回退到默认安全提醒 + 基础高风险置顶版本。
- 如果 UI 优化影响主流程可用性，先回退到阶段一可跑通版本。
- 如果部署版本不稳定，优先保留本地稳定演示环境，保证比赛展示不断档。
- 如果 AI 输出结构不稳定，优先缩减输出自由度，保留最基础字段：通俗解释、步骤卡、安全提醒。

## 11. 真实 AI 验证说明

### 11.1 需要配置的环境变量

真实 AI 成功路径验证时，后端至少需要配置以下环境变量：

- `OPENAI_API_KEY`：OpenAI 兼容接口的密钥。
- `OPENAI_BASE_URL`：AI 服务基础地址。
- `OPENAI_MODEL`：模型名称。

如果后续接入阿里云百炼的免费 key，也可以使用兼容的百炼配置方式：

- `DASHSCOPE_API_KEY`：百炼密钥。
- `OPENAI_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1`
- `OPENAI_MODEL=qwen-plus`

当前后端实现会优先读取 `OPENAI_API_KEY`，同时兼容 `DASHSCOPE_API_KEY`，便于后续切换模型服务。

### 11.2 本地启动命令示例

在配置好环境变量后，可使用下面的方式启动：

```bash
npm start
```

如果需要临时设置环境变量，可以在启动前按当前系统方式配置后再执行 `npm start`。

### 11.3 需要验证的 4 个用例

1. **普通医保说明**
   - 输入一段正常医保说明。
   - 预期返回 `success: true`。
   - `data` 中应包含通俗解释、步骤卡、安全提醒、`riskLevel`、`riskTags`。

2. **银行高风险验证码**
   - 输入包含“验证码”或类似高风险词的银行说明。
   - 预期返回 `success: true`。
   - `riskLevel` 应为 `high`。
   - `riskTags` 应包含对应风险词。
   - `priorityReminder` 应置顶强调不要直接继续。

3. **AI 返回结构异常**
   - 模拟 AI 返回非 JSON、缺字段、空内容或异常结构。
   - 预期后端返回 `success: false`。
   - `error.code` 应明确，例如 `AI_RESPONSE_INVALID`、`AI_RESPONSE_EMPTY` 或 `AI_REQUEST_FAILED`。

4. **空输入**
   - 不填写说明文本直接提交。
   - 预期返回 `success: false`。
   - `error.code` 应为 `TEXT_REQUIRED`。

### 11.4 验证原则

- 真实 AI 成功路径必须看见 `success: true`，不能由 fallback 假成功代替。
- 失败路径必须真实返回 `success: false + error`。
- 如果当前没有可用 Key，就只保留这份验证说明，不进入 UI 优化。
