# 安心办 MVP UI 文档草案

## 1. 设计目标

安心办的界面设计必须围绕“老人本人使用”展开，核心不是追求丰富功能，而是帮助老人把复杂说明看懂、按顺序操作、在风险点停下来确认。

第一版 UI 只服务一条主流程：选择场景 → 粘贴说明 → 查看安全结论 → 查看通俗解释 → 查看步骤卡。

界面设计优先满足以下目标：

- 大字，保证老人能直接看清主要内容。
- 高对比，保证文字、按钮、提醒信息一眼能分清。
- 少按钮，避免页面上同时出现过多操作。
- 移动端优先，默认按手机单手浏览场景设计。
- 一步一屏，让用户每次只处理一个核心任务。
- 风险优先，在高风险内容出现时先提醒，再引导继续阅读。

## 2. 页面结构

第一版 MVP 建议控制为 4 个核心页面，每个页面只承担一个主要任务。

### 2.1 首页：场景选择页

首页用于帮助用户先判断“我现在要处理的是哪一类说明”。

页面应包含：

- 页面标题：安心办
- 简短说明：帮你把复杂说明看懂
- 三个场景入口卡片：医保、银行、政务
- 页面底部简短安全提示

首页三个场景入口在 UI 上保持平权：

- 三个入口尺寸一致
- 三个入口层级一致
- 三个入口视觉权重一致
- 不通过大小、颜色、排序暗示某一个场景更重要

每张场景卡片采用统一结构：图标 + 场景名称 + 一句短说明。

其中：

- 小图标只做辅助识别
- 场景名称必须比图标更醒目
- 一句短说明只负责补充理解
- 不使用复杂插画，避免分散注意力

但在首发演示内容准备上，优先使用医保案例作为主示例，用于讲清产品价值和主流程。

#### 2.1.1 通用兜底入口

在三个主场景卡片下方，提供一个辅助兜底入口，用于老人无法判断说明属于哪一类场景时使用。

入口结构：

- 标题：不知道选哪个？
- 说明：看不懂的 App 说明，也可以先放这里
- 按钮语义：直接帮我看看

设计要求：

- 视觉权重弱于三个主场景卡片（使用虚线边框、更小字号、更浅文字色）。
- 不喧宾夺主，让老人优先看到三个主场景。
- 点击后进入与主场景相同的说明输入页，标题显示"通用场景说明输入"。
- 后端对应 `general` 场景，AI 输出结构与其他场景一致。

### 2.2 说明输入页

该页面只负责让用户粘贴公开说明文本。

页面应包含：

- 当前场景标题
- 返回上一步入口
- 说明输入框
- 输入说明文字的简短引导
- 生成按钮
- 默认显示的简短安全提醒

页面重点是让用户知道“只需要把公开说明粘贴进来”，不要让用户以为需要填写账号、证件、验证码等信息。

输入框状态规则：

- 从一个场景切换到另一个场景时，输入框应清空，避免医保内容残留到银行、政务或通用场景。
- 从结果页返回修改同一场景时，可以保留原输入，方便用户只改错别字或补充说明。
- 输入内容去掉前后空白后少于 10 个字符时，应提示“这段太短了，多贴几句完整说明。”。

### 2.3 解释与步骤页

该页面是 MVP 的核心阅读页面，用于展示 AI 输出内容。

页面应包含：

- 安全结论卡区域（顶部，先给"能不能继续操作"的结论）
- 通俗解释区域
- 步骤卡区域
- 上一步 / 下一步按钮
- 当前步骤位置提示
- 高风险提醒区域（仅 high 场景显示）

页面重点是帮助老人先得到结论，再理解，再行动，因此信息顺序应遵循：

普通场景下：先看结论卡 → 再看解释 → 再看步骤

当内容属于高风险场景时，结论卡显示红色"先别点"，高风险提醒条和确认按钮同时置顶展示，先提醒，再展示解释和步骤。

普通场景中，首屏应优先展示"安全结论卡"和"通俗解释"，帮助老人先明白能不能继续、这段话是什么意思；随后再进入"步骤卡"阅读。步骤卡仍然采用一屏一步方式展示。

### 2.4 错误与空状态页

该页面用于处理用户没有输入内容、内容无法理解、生成失败等情况。

页面应包含：

- 明确错误标题
- 一句简单原因说明
- 一条下一步建议
- 返回或重新输入按钮

错误页的重点不是解释技术问题，而是告诉老人“现在怎么办”。

## 3. 主流程

### 3.1 正常流程

1. 用户进入首页。
2. 用户选择医保、银行或政务场景。
3. 用户进入说明输入页，粘贴公开说明文本。
4. 用户点击生成。
5. 用户进入解释与步骤页，先看到安全结论卡和通俗解释。
6. 用户逐步查看步骤卡。

### 3.2 高风险流程

当文本中出现银行、验证码、转账、陌生链接、收费等高风险内容时：

1. 用户进入结果页后，先看到红色安全结论卡和加强版高风险提醒。
2. 高风险提醒位置优先于通俗解释和步骤卡。
3. 提醒语气保持保守，明确建议先联系官方渠道、家人或可信工作人员确认。
4. 提醒区域提供两个明确动作按钮：“先找家人确认”和“联系官方渠道”。
5. 这两个按钮只承担提醒和引导作用，不跳转真实服务，不收集任何信息。

### 3.3 老人使用流程原则

- 每一步只做一件事。
- 每个页面只保留一个主操作。
- 每个主操作都用容易理解的文字，不用复杂术语。
- 重要信息放前面，次要信息放后面。

## 4. 组件说明

### 4.1 场景卡片

用于首页的三个场景入口。

设计要求：

- 点击区域大
- 文案短
- 易识别
- 三张卡片风格统一

卡片上采用“图标 + 场景名称 + 一句短说明”的固定结构，不放过多说明文字。

其中图标只做辅助识别，文字始终是主信息。

### 4.2 说明输入框

用于粘贴公开说明文本。

设计要求：

- 输入区域足够大
- 默认提示语清楚
- 不出现需要填写敏感信息的暗示
- 附近直接显示安全提醒短句

### 4.3 主操作按钮

用于“下一步”“生成”“返回”“重新输入”等关键操作。

设计要求：

- 页面中同一时刻只突出一个主按钮
- 按钮文案直接表达动作
- 按钮尺寸适合老人点击

### 4.4 通俗解释卡片

用于展示 AI 对原始说明的简化解释。

设计要求：

- 文字分段清晰
- 句子不要太长
- 不单独再做“原文要点”模块
- 优先回答“这段话在说什么”

### 4.5 步骤卡

用于逐步展示操作流程。

设计要求：

- 一张卡只讲一个动作
- 当前步骤足够醒目
- 上一步 / 下一步位置固定
- 当前步骤不要被其他信息打断

### 4.6 高风险提醒框

仅用于 high 场景，展示加强版风险提醒和确认动作。

设计要求：

- 普通和中风险场景不展示底部安全提醒卡，避免重复信息。
- 高风险时样式醒目，位置在结论卡下方、通俗解释上方。
- 文案保守、直接、易懂。
- 高风险时增加两个明确动作按钮：“先找家人确认”“联系官方渠道”。
- 动作按钮只做提醒引导，不跳转真实业务，不收集信息。

### 4.7 安全结论卡

用于在结果页顶部先告诉老人“这件事能不能继续办”。

设计要求：

- 所有结果页都显示，不只在高风险时显示。
- low 显示绿色“可以继续办”，medium 显示琥珀色“先看清楚，再继续”，high 显示红色“先别点，可能有风险”。
- 图标使用内联 SVG，文字仍然是主信息，不能只靠颜色或图标传达风险。
- high 档时，结论卡下方继续显示高风险提醒条，不互相替代。
- 移动端保持图标和文字不挤压，标题不能换成难懂的技术词。

## 5. 状态设计

### 5.1 默认状态

- 首页默认展示三个平权场景入口
- 输入页默认展示空输入框和简短安全提示
- 结果页默认先展示安全结论卡，再展示通俗解释和步骤卡

### 5.2 输入为空状态

- 明确提示“请先粘贴说明内容”
- 不使用复杂报错语言
- 给出清晰下一步动作

### 5.3 生成中状态

- 明确告诉用户“正在帮你整理说明”
- 页面内容保持简单，不制造焦虑
- 不同时展示过多变化元素

### 5.4 生成失败状态

- 直接说明“暂时没整理成功”
- 给出重新尝试或返回修改内容的入口
- 不展示技术原因和复杂错误代码

### 5.5 高风险提醒状态

- 当识别到高风险内容时，安全结论卡显示红色“先别点，可能有风险”
- 高风险提醒条显示在结论卡下方，继续优先于通俗解释
- 提醒内容比默认状态更明确
- 页面其他内容仍可查看，但视觉重点先给提醒
- 高风险提醒中显示两个明确动作按钮，帮助老人先停下来确认

## 6. 无障碍要求

- 主要文字必须足够大，避免老人需要频繁放大页面。
- 文本和背景必须有明显对比，不能依赖低对比浅色文字。
- 关键信息不能只靠颜色区分，必须同时有文字说明。
- 按钮点击区域要大，避免误触。
- 页面信息分组清楚，避免长段连续文字。
- 每个页面主标题明确，让老人知道自己当前在哪一步。
- 高风险提醒必须易看见，不能藏在页面底部。

## 7. 移动端适配

安心办第一版默认按手机端设计，页面应优先适配竖屏移动端浏览。

移动端设计要求：

- 单列布局，不做复杂多栏。
- 重要内容从上到下顺序排列。
- 主按钮放在容易点击的位置。
- 页面滚动距离尽量短。
- 步骤切换操作清楚，避免误触。
- 首屏尽量先展示当前最重要的信息。

在结果页中：

- 普通场景下，首屏优先看到安全结论卡和通俗解释。
- 高风险场景下，首屏优先看到红色结论卡、高风险提醒和确认按钮。

## 8. 不要做什么

- 不做复杂导航层级。
- 不做多入口并列的功能堆叠。
- 不做过多颜色和装饰效果，避免分散注意力。
- 不做小字密集排版。
- 不做一页展示过多按钮。
- 不做需要老人理解复杂术语的说明结构。
- 不做登录、历史记录、OCR、语音输入等非 MVP 页面。
- 不做引导用户填写身份证号、银行卡号、密码、验证码、支付信息、真实医保账号等敏感信息的界面。
- 不把银行、医保、政务做成不同风格过强的三套页面，避免增加学习成本。
