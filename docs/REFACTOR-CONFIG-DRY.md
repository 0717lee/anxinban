# 前后端共享配置重构评估报告与行动计划

## 1. 背景

本报告针对当前代码中的前后端共享配置重复问题进行评估，并给出可执行的重构行动计划。

当前项目是“安心办”MVP，技术路线保持轻量：

- 前端为静态单页 Web。
- 后端为 Node.js 轻量 HTTP 服务。
- 主链路为：选择场景 → 粘贴说明 → AI 生成通俗解释 / 步骤卡 / 安全提醒 → 前端展示。
- 产品边界要求不接入真实业务系统，不收集敏感信息，不引入复杂后台。

在这个阶段，配置类逻辑应尽量集中管理，避免前后端在快速迭代中出现规则不一致。

## 2. 问题描述

### 2.1 重复定义的场景标签映射

`getSceneLabel` 在前后端重复定义，逻辑完全一致。

后端位置：`server/index.js`

```javascript
function getSceneLabel(scene) {
  if (scene === 'medical') return '医保';
  if (scene === 'bank') return '银行';
  return '政务';
}
```

前端位置：`web/app.js`

```javascript
function getSceneLabel(scene) {
  if (scene === 'medical') return '医保';
  if (scene === 'bank') return '银行';
  return '政务';
}
```

这说明“场景代码到中文标签”的映射规则没有单一来源。后续如果新增 `telecom`、`insurance` 等场景，或调整“政务”为“政务服务”，需要同时修改前端和后端，容易遗漏。

### 2.2 风险关键词列表只在后端定义

`RISK_KEYWORDS` 当前定义在 `server/index.js`：

```javascript
const RISK_KEYWORDS = ['验证码', '密码', '银行卡', '转账', '付款', '收费', '链接', '陌生电话', '账户冻结'];
```

该列表承担安全兜底职责，是风险识别规则的一部分。虽然当前前端没有直接引用，但产品链路中前端需要展示安全提醒，后续很可能需要基于同一组关键词做输入前提示、风险预提醒或本地高亮。

如果前端未来也手写一份相同列表，会形成第二个重复源，导致风险判断规则分叉。

### 2.3 配置和业务逻辑耦合

当前配置散落在业务文件中：

- 场景标签逻辑写在前后端主流程文件内。
- 风险关键词写在后端入口文件内。
- 场景合法性校验写成独立数组：`['medical', 'bank', 'government']`。

这些配置都属于“业务常量”，但目前没有统一组织，导致新增场景时至少要检查三类位置：展示标签、合法性校验、提示词语义。

## 3. 影响评估

### 3.1 维护成本

每次调整场景或风险关键词，都需要人工搜索和同步多处代码。项目越快迭代，这种同步成本越高。

典型风险包括：

- 前端显示了新场景，但后端校验不通过。
- 后端支持了新场景，但前端标题仍展示默认“政务”。
- 后端风险提示词更新了，前端本地提示仍使用旧规则。

### 3.2 一致性风险

该项目面向老人理解复杂说明，安全提醒是核心价值之一。风险关键词如果前后端不一致，会造成同一段文本在不同环节得到不同提示，影响用户信任。

### 3.3 扩展风险

当前 MVP 只有医保、银行、政务三个场景，因此重复代码看起来成本较低。但从技术文档和开发计划看，后续很可能围绕高风险处理、UI 优化、演示强化继续迭代。此时共享配置如果不提前收敛，会影响后续扩展效率。

## 4. 重构目标

本次重构目标不是引入复杂配置系统，而是在不破坏 MVP 轻量架构的前提下，建立明确的共享配置来源。

目标如下：

- 消除 `getSceneLabel` 的前后端重复实现。
- 让场景列表、场景标签、风险关键词拥有统一维护入口。
- 保持前端静态页和轻量后端结构，不引入构建工具。
- 保持 `/api/generate` 主链路行为不变。
- 为后续前端本地风险提示、高亮或输入校验预留配置来源。

## 5. 方案对比

### 5.1 方案一：新增后端配置接口

后端集中定义应用配置，并通过 `/api/config` 返回给前端。

建议配置结构：

```javascript
const APP_CONFIG = {
  scenes: {
    medical: '医保',
    bank: '银行',
    government: '政务'
  },
  riskKeywords: ['验证码', '密码', '银行卡', '转账', '付款', '收费', '链接', '陌生电话', '账户冻结']
};
```

前端启动时请求 `/api/config`，并使用返回的 `scenes` 做场景标题展示。

优点：

- 单一来源清晰，后端掌握配置权威版本。
- 不需要引入构建工具。
- 适合当前“静态前端 + 轻量后端”架构。
- 后续可以扩展更多运行时配置。

缺点：

- 前端首次进入页面多一次请求。
- 需要处理配置接口失败时的降级逻辑。

### 5.2 方案二：新增共享 JSON 静态文件

在项目中新增类似 `shared/config.json` 或 `web/config.json` 的配置文件，后端读取文件，前端通过静态路径请求同一份 JSON。

优点：

- 配置天然是单独文件，容易查看和修改。
- 前后端都可以读取同一份数据。
- 不需要模块打包。

缺点：

- 后端需要处理文件读取和解析异常。
- 静态暴露配置文件时需要明确只包含可公开数据。
- 当前静态文件服务需要确保路径安全和 MIME 类型处理。

### 5.3 方案三：引入前后端共享模块

创建 `shared/config.js`，后端通过 `require` 使用，前端通过 `<script>` 引入。

优点：

- 代码层面共享更直接。
- 无需前端异步请求配置。

缺点：

- 浏览器环境和 Node.js CommonJS 共享同一文件需要额外兼容写法。
- 容易出现全局变量污染。
- 对当前无构建工具的静态前端来说，可维护性不如配置接口或 JSON 文件。

## 6. 推荐方案

推荐采用“方案一：新增后端配置接口”。

理由如下：

- 与当前后端已经承担 AI 调用、输入校验、风险兜底的职责一致。
- 不需要引入构建工具或改变部署形态。
- 可以在后端保持配置单一来源，前端只消费配置结果。
- 配置接口天然适合后续扩展，例如风险提示文案、场景描述、动作按钮文案等。

推荐实施时保持克制，只做必要能力：

- 不做复杂缓存版本管理。
- 不做后台管理配置。
- 不做远程动态配置中心。
- 不把任何密钥、模型参数或敏感服务配置暴露给前端。

## 7. 目标设计

### 7.1 后端配置结构

在 `server/index.js` 顶部集中定义公开配置：

```javascript
const APP_CONFIG = {
  scenes: {
    medical: '医保',
    bank: '银行',
    government: '政务'
  },
  riskKeywords: ['验证码', '密码', '银行卡', '转账', '付款', '收费', '链接', '陌生电话', '账户冻结']
};
```

后端内部继续提供 `getSceneLabel(scene)`，但实现改为读取 `APP_CONFIG.scenes`：

```javascript
function getSceneLabel(scene) {
  return APP_CONFIG.scenes[scene] || APP_CONFIG.scenes.government;
}
```

`isValidScene(scene)` 也应改为基于 `APP_CONFIG.scenes` 判断：

```javascript
function isValidScene(scene) {
  return Object.prototype.hasOwnProperty.call(APP_CONFIG.scenes, scene);
}
```

这样新增场景时，不需要再维护单独的合法场景数组。

### 7.2 配置接口

新增只读接口：

```text
GET /api/config
```

返回结构：

```json
{
  "success": true,
  "data": {
    "scenes": {
      "medical": "医保",
      "bank": "银行",
      "government": "政务"
    },
    "riskKeywords": ["验证码", "密码", "银行卡", "转账", "付款", "收费", "链接", "陌生电话", "账户冻结"]
  },
  "error": null
}
```

说明：

- 建议沿用项目已有的 `success + data + error` 响应外壳。
- 该接口只返回可公开配置，不返回任何 AI 密钥、模型名、Base URL 或超时参数。
- 可以为响应添加 `Cache-Control: no-store` 或短缓存策略；MVP 阶段优先简单可靠。

### 7.3 前端配置加载

前端新增默认配置和运行时配置变量：

```javascript
const DEFAULT_APP_CONFIG = {
  scenes: {
    medical: '医保',
    bank: '银行',
    government: '政务'
  },
  riskKeywords: ['验证码', '密码', '银行卡', '转账', '付款', '收费', '链接', '陌生电话', '账户冻结']
};

let appConfig = DEFAULT_APP_CONFIG;
```

页面初始化时加载后端配置，失败时使用默认配置兜底。

前端 `getSceneLabel(scene)` 保留为薄封装，但不再写死分支逻辑：

```javascript
function getSceneLabel(scene) {
  return appConfig.scenes[scene] || DEFAULT_APP_CONFIG.scenes.government;
}
```

这个函数此时不再是重复业务规则，而是前端读取配置的展示辅助函数。

### 7.4 初始化顺序

前端当前脚本直接在顶层绑定事件。重构时建议保持结构简单：

1. 顶层声明 DOM 引用、状态、默认配置。
2. 定义 `loadAppConfig()`。
3. 定义 `bindEvents()`。
4. 调用 `initApp()`，先加载配置，再绑定事件。

这样可以避免用户在配置未加载完成前点击场景导致标题使用旧配置。

## 8. 行动计划

### 8.1 阶段一：收敛后端配置

目标：让后端场景和风险关键词拥有统一配置入口。

任务：

- 在 `server/index.js` 中新增 `APP_CONFIG`。
- 将 `RISK_KEYWORDS` 合并到 `APP_CONFIG.riskKeywords`。
- 将 `getSceneLabel(scene)` 改为读取 `APP_CONFIG.scenes`。
- 将 `isValidScene(scene)` 改为读取 `APP_CONFIG.scenes` 的 key。
- 确认 `buildPrompt(scene, text)` 仍使用正确中文场景标签。

验收标准：

- `/api/generate` 原有请求和响应结构不变。
- 医保、银行、政务三个场景仍能通过校验。
- 非法场景仍被拒绝。
- 高风险关键词兜底逻辑仍引用同一组关键词。

### 8.2 阶段二：新增配置接口

目标：让前端可以从后端获取公开配置。

任务：

- 新增 `GET /api/config` 路由。
- 返回 `success + data + error` 响应外壳。
- `data` 中包含 `scenes` 和 `riskKeywords`。
- 确保接口不暴露环境变量、AI 密钥、模型名或其他敏感运行时配置。

验收标准：

- 浏览器访问 `/api/config` 能得到 JSON。
- 返回结构与技术文档中的统一响应外壳一致。
- 配置内容与后端内部使用的配置一致。

### 8.3 阶段三：改造前端配置消费

目标：移除前端写死的场景映射分支。

任务：

- 在 `web/app.js` 中新增默认配置和运行时配置。
- 新增 `loadAppConfig()` 请求 `/api/config`。
- 将前端 `getSceneLabel(scene)` 改为读取 `appConfig.scenes`。
- 将事件绑定收敛到 `bindEvents()`，由 `initApp()` 统一启动。
- 保留配置加载失败时的默认配置兜底。

验收标准：

- 前端不再出现与后端相同的 `if scene === ...` 分支映射。
- 配置接口正常时，场景标题来自后端配置。
- 配置接口失败时，页面仍可使用默认配置完成主链路。

### 8.4 阶段四：补充验证

目标：确认重构没有破坏 MVP 主链路。

任务：

- 启动服务并访问首页。
- 检查 `/api/config` 返回内容。
- 分别点击医保、银行、政务场景，确认标题正确。
- 使用一段普通公开说明测试 `/api/generate`。
- 使用包含“验证码”“银行卡”“转账”的文本测试风险兜底。

验收标准：

- 首页正常加载。
- 场景选择正常跳转。
- AI 结果展示流程不回退。
- 高风险安全提醒仍正常触发。

## 9. 回归测试范围

### 9.1 配置接口测试

测试点：

- `GET /api/config` 返回 HTTP 200。
- 响应为 JSON。
- 响应包含 `success: true`、`data.scenes`、`data.riskKeywords`、`error: null`。
- `data.scenes.medical` 为“医保”。
- `data.scenes.bank` 为“银行”。
- `data.scenes.government` 为“政务”。
- `data.riskKeywords` 包含“验证码”“密码”“银行卡”“转账”。

### 9.2 场景选择测试

测试点：

- 点击医保卡片，标题显示“医保场景说明输入”。
- 点击银行卡片，标题显示“银行场景说明输入”。
- 点击政务卡片，标题显示“政务场景说明输入”。
- 返回上一步后重新选择场景，标题能正确更新。

### 9.3 生成接口测试

测试点：

- 合法场景 `medical`、`bank`、`government` 均能进入生成流程。
- 非法场景会返回明确错误。
- 空文本会被拒绝。
- 超长文本会被请求体大小限制拦截。
- AI 返回异常时仍能展示可理解错误。

### 9.4 风险关键词测试

测试点：

- 输入包含“验证码”的文本，应触发风险兜底。
- 输入包含“银行卡”的文本，应触发风险兜底。
- 输入包含“转账”的文本，应触发风险兜底。
- 普通低风险公开说明不应被误判为高风险。

### 9.5 降级测试

测试点：

- 配置接口异常时，前端使用默认配置。
- 配置接口返回格式异常时，前端使用默认配置。
- 配置加载失败不影响用户选择场景和进入输入页。

## 10. 风险与控制措施

### 10.1 配置接口失败导致前端不可用

控制措施：前端保留 `DEFAULT_APP_CONFIG`。配置接口失败时，只影响动态配置更新，不影响 MVP 主链路。

### 10.2 公开配置接口误暴露敏感信息

控制措施：`/api/config` 只返回白名单字段：`scenes` 和 `riskKeywords`。不得返回环境变量、AI 配置、密钥、模型名、Base URL。

### 10.3 初始化顺序改变引入事件绑定问题

控制措施：将事件绑定集中在 `bindEvents()`，由 `initApp()` 调用；避免在多个位置重复绑定点击事件。

### 10.4 默认配置再次形成重复

控制措施：默认配置仅作为前端容灾兜底，不作为常规维护入口。正式修改以 `APP_CONFIG` 为准，并在文档中说明这一点。

## 11. 建议实施顺序

建议按以下顺序实施：

1. 先重构后端 `APP_CONFIG`，确保后端内部配置统一。
2. 再新增 `/api/config`，保持接口只读和白名单输出。
3. 再改造前端初始化流程，消费配置接口。
4. 最后做主链路回归测试和风险关键词测试。

不建议先改前端，因为前端依赖的配置来源尚未稳定；也不建议一次性引入共享模块或构建工具，因为会超出当前 MVP 的轻量技术边界。

## 12. 完成定义

本次重构完成时，应满足以下条件：

- `server/index.js` 中只有一个权威配置对象维护场景和风险关键词。
- `web/app.js` 不再硬编码场景到中文标签的分支映射。
- `/api/config` 能稳定返回公开配置。
- `/api/generate` 行为保持兼容。
- README 或技术文档后续可补充“公开配置来源”的说明。
- 回归测试覆盖配置接口、场景选择、生成接口、风险兜底和降级行为。

## 13. 后续扩展建议

完成本次重构后，可以在同一配置结构中逐步纳入更多公开配置：

- 场景说明文案，例如“医保说明更容易看懂”。
- 场景图标，例如“🏥”“🏦”“🏛️”。
- 安全动作按钮文案，例如“先找家人确认”“联系官方渠道”。
- 不同场景下的默认输入示例。
- 前端本地风险预提醒文案。

这些扩展不应在本次重构中一次性完成，避免扩大范围。当前最优先的是先消除已确认的重复配置问题，并保证主链路稳定。
