# 安心办 MVP 技术文档草案

## 1. 技术目标

安心办第一版 MVP 的技术目标，不是建设复杂系统，而是用最小可行的技术方案稳定跑通这条主路径：

选择场景 → 粘贴说明 → AI 生成通俗解释 / 步骤卡 / 安全提醒 → 前端展示

技术方案必须优先满足以下原则：

- 主流程简单清晰，便于初赛演示。
- 前后端职责明确，方便后续扩展。
- AI 输出结构固定，方便前端直接展示。
- 不收集敏感信息，不接入真实业务接口。
- 错误可理解，失败时能明确提示用户下一步。
- 部署方式简单，适合快速上线演示版本。

## 2. MVP 技术边界

第一版 MVP 只处理“公开说明文本的理解与展示”，不处理真实业务办理。

### 2.1 本期必须支持

- 前端提供场景选择页面。
- 前端提供说明文本输入页面。
- 后端接收场景和公开说明文本。
- 后端调用 AI 生成结构化结果，并使用统一响应外壳返回前端。
- 前端展示通俗解释、步骤卡、安全提醒。
- 前端根据高风险内容调整安全提醒展示优先级。

统一响应外壳要求为 `success + data + error`：

- `success` 表示本次请求是否成功。
- `data` 承载 AI 返回的固定 JSON 结构。
- `error` 承载失败时的错误码和错误说明。

### 2.2 本期明确不做

- 不做登录注册。
- 不做历史记录。
- 不做 OCR 拍照识别。
- 不做语音输入。
- 不接入真实医保、银行、政务接口。
- 不做真实业务跳转。
- 不收集身份证号、银行卡号、密码、验证码、支付信息、真实医保账号等敏感信息。

## 3. 系统角色划分

### 3.1 前端职责

前端只负责四类工作：

- 让用户选择场景。
- 让用户粘贴公开说明文本。
- 调用后端生成结果。
- 把结构化结果按 UI 文档要求展示出来。

前端不负责解释业务规则，不直接拼接复杂 AI 提示内容，也不判断真实业务是否可办理。

### 3.2 后端职责

后端只负责四类工作：

- 接收前端提交的场景和说明文本。
- 做基础输入校验。
- 调用 AI 服务。
- 把 AI 返回结果整理成固定结构，返回前端。

后端还负责把最终结果包装成统一响应外壳，方便前端统一处理成功、失败和异常状态。

后端不保存长期用户资料，不处理真实业务账户，不代替用户执行任何外部操作。

### 3.3 AI 服务职责

AI 服务只负责理解用户粘贴的公开说明文本，并输出固定结构的辅助理解结果，包括：

- 通俗解释
- 步骤卡
- 安全提醒
- 风险等级与风险标签

AI 服务不应输出“替用户办理”的结论，不应要求补充敏感信息，也不应引导真实转账、支付或验证码操作。

### 3.4 响应外壳

后端返回给前端的结果统一采用：

```json
{
  "success": true,
  "data": {
    "scene": "medical",
    "plainExplanation": {},
    "steps": [],
    "riskLevel": "low",
    "riskTags": [],
    "safety": {}
  },
  "error": null
}
```

失败时返回：

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "AI_RESPONSE_INVALID",
    "message": "暂时没整理成功，请稍后再试。"
  }
}
```

其中 `data` 内部仍然沿用固定 JSON 结构。

## 4. 前端方案

第一版前端建议采用单页式移动端 Web 界面，原因很简单：

- 适合比赛演示。
- 适合手机访问。
- 页面切换和结果展示更直接。
- 后续部署简单。

前端只需要覆盖以下页面能力：

- 场景选择页
- 说明输入页
- 解释与步骤页
- 错误与空状态页

前端核心处理逻辑：

- 记录用户当前选择的场景。
- 提交说明文本到后端。
- 接收统一响应外壳中的结构化 AI 结果。
- 根据 `riskLevel` 和后端关键词兜底结果决定默认展示还是高风险置顶展示。
- 将步骤卡按一屏一步方式切换显示。

## 5. 后端方案

第一版后端建议采用轻量 API 服务，只提供少量接口，重点是简单、稳定、可维护。

后端核心任务：

- 接收请求
- 校验参数
- 调用 AI
- 标准化返回结果
- 处理错误

从第一性原理看，这个项目的后端不是“业务系统”，而是“AI 结果整理层”。

也就是说，后端最重要的价值不是保存数据，而是把不稳定的 AI 输出，尽量整理成前端可直接消费的稳定结构。

## 6. AI 调用方式

### 6.1 调用目标

AI 调用的目标不是生成长篇回答，而是生成适合前端展示的固定结构结果。

因此，AI 输入应至少包含：

- 场景类型：医保 / 银行 / 政务 / 通用（general）
- 用户粘贴的公开说明文本
- 输出规则：必须返回通俗解释、步骤卡、安全提醒、风险等级和风险标签
- 安全规则：不得要求用户提供敏感信息，不得引导高风险真实操作

其中 `general` 场景作为兜底入口，用于老人无法判断说明属于哪一类场景时使用。AI 在 `general` 场景下仍然输出固定结构，安全风险判断标准与其他场景一致，不因通用场景放宽风险阈值。

### 6.2 调用原则

- 提示词应尽量稳定，不因前端页面变化频繁修改。
- 输出结构必须清晰，避免前端再做复杂文本解析。
- 高风险内容必须显式标记，不能只靠长段文字隐含表达。
- AI 返回失败时，后端必须能识别并返回明确错误状态。

风险判定采用 AI + 后端窄规则兜底：

- 第一层：AI 识别为主，输出 `riskLevel` 和 `riskTags`。
- 第二层：后端高风险关键词兜底，遇到验证码、密码、银行卡、转账、付款、收费、链接、陌生电话、账户冻结等词时升为 high。
- 第三层：后端注意事项关键词兜底，只有在 AI 返回 low 且没有高风险词时，遇到证件原件、到社区、窗口、现场核对、本人信息、提交材料、等待审核、线下办理、预约等词时升为 medium。

这样做的原因很简单：AI 更灵活，关键词更稳定，窄规则只补足明确场景，减少漏标和误报。

## 7. AI 输出结构

为了方便前端展示，AI 输出建议固定为结构化结果。可以理解为：不是让 AI 写一篇文章，而是让 AI 按固定表单填空。

建议结构如下：

```json
{
  "success": true,
  "data": {
    "scene": "medical",
    "plainExplanation": {
      "title": "这段说明在讲什么",
      "summary": "这是对原始说明的通俗解释。",
      "tips": [
        "用户需要先确认自己当前要办理的事项。",
        "说明里有些内容只是提示，不一定都要立即操作。"
      ]
    },
    "steps": [
      {
        "stepNumber": 1,
        "title": "先找到对应入口",
        "content": "打开相关页面后，先找到这项服务入口。"
      },
      {
        "stepNumber": 2,
        "title": "按提示查看信息",
        "content": "先阅读页面提示，再决定是否继续。"
      }
    ],
    "riskLevel": "medium",
    "riskTags": ["official_confirmation_needed"],
    "safety": {
      "defaultReminder": "遇到不确定内容，先确认再操作。",
      "priorityReminder": "如果涉及收费、验证码或转账，不要直接继续，请先联系官方渠道或家人确认。",
      "actions": [
        "先找家人确认",
        "联系官方渠道"
      ]
    }
  },
  "error": null
}
```

### 7.1 字段说明

- `success`：是否成功。
- `data`：成功时的业务结果。
- `error`：失败时的错误对象。
- `data.scene`：当前场景类型，用于前端展示与校验。
- `plainExplanation`：通俗解释模块。
- `plainExplanation.title`：解释区标题。
- `plainExplanation.summary`：一段核心解释，回答“这段话在说什么”。
- `plainExplanation.tips`：补充提醒，使用短句数组，方便分点展示。
- `steps`：步骤卡数组，每一项只表达一个动作。
- `riskLevel`：风险等级，建议限定为 `low / medium / high`。
- `riskTags`：风险标签数组，用于前端识别高风险内容类型。
- `safety.defaultReminder`：AI 生成的基础安全提醒，当前不在普通结果页单独成卡展示，避免和安全结论卡重复。
- `safety.priorityReminder`：高风险时优先展示的加强提醒。
- `safety.actions`：提醒动作按钮文案，仅用于引导，不触发真实业务操作。

### 7.2 前端展示规则

- 普通场景下，前端优先展示 `plainExplanation`，再展示 `steps`。
- 高风险场景下，当前端看到 `riskLevel = high` 或存在高风险 `riskTags` 时，应优先展示 `safety.priorityReminder`。
- `steps` 必须按顺序展示，支持一屏一步切换。
- `safety.actions` 只作为提醒动作按钮，不绑定真实跳转。

## 8. 数据流

### 8.1 正常数据流

1. 用户在前端选择场景。
2. 用户粘贴公开说明文本。
3. 前端将 `scene + text` 提交给后端。
4. 后端做基础校验。
5. 后端调用 AI 服务。
6. AI 返回结构化结果。
7. 后端校验返回结构是否完整。
8. 后端包装为 `success + data + error` 响应外壳。
9. 前端按 UI 规则展示安全结论卡、通俗解释和步骤卡；高风险时额外展示高风险提醒。

### 8.2 高风险数据流

1. AI 在输出中标记 `riskLevel` 和 `riskTags`。
2. 后端先使用高风险关键词规则兜底，必要时升为 high。
3. 若无高风险词且 AI 返回 low，再使用注意事项关键词规则兜底，必要时升为 medium。
4. 后端合并风险结果后返回前端。
5. 前端根据 `riskLevel` 展示安全结论卡。
6. 当为 high 时，继续优先显示加强版高风险提醒和确认动作按钮。

## 9. 错误处理

第一版 MVP 的错误处理原则是：不要把系统内部复杂性暴露给老人，只告诉他现在发生了什么、下一步该怎么做。

### 9.1 前端错误场景

- 未选择场景
- 输入内容为空
- 输入内容过短或明显无效

输入内容过短的统一判定：前后端均先对文本执行 `trim()`，去掉前后空白。去掉空白后长度为 0 时返回 `TEXT_REQUIRED`；长度为 1-9 个字符时返回 `TEXT_TOO_SHORT`。中文汉字按 JavaScript 字符长度计数，普通汉字通常 1 个字计 1 个字符。
- 请求超时
- 结果生成失败

前端错误提示要尽量简单，例如：

- 请先选择场景
- 请先粘贴说明内容
- 这段内容太短，建议补充完整说明
- 暂时没整理成功，请稍后再试

### 9.2 后端错误场景

- 请求参数缺失
- 输入文本不合法
- AI 服务调用失败
- AI 返回结构不完整
- 服务内部异常

后端对前端返回时，建议统一为易理解的错误结构，避免直接暴露底层细节。

建议结构如下：

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "AI_RESPONSE_INVALID",
    "message": "暂时没整理成功，请稍后再试。"
  }
}
```

## 10. 安全边界

### 10.1 输入安全边界

- 只允许输入公开说明文本。
- 不主动要求身份证号、银行卡号、密码、验证码、支付信息、真实医保账号。
- 如果文本中疑似出现敏感信息，系统应提醒用户删除后再试。

### 10.2 输出安全边界

- 输出只能做“解释”和“提醒”。
- 不输出“请立即转账”“请输入验证码”“点击此链接继续”等高风险引导。
- 不输出替代官方判断的确定性承诺。

### 10.3 高风险内容处理

当说明文本涉及以下内容时，应识别为重点风险内容：

- 验证码
- 密码
- 银行卡
- 转账
- 付款
- 收费
- 链接
- 陌生电话
- 账户冻结
- 要求提供身份证、银行卡、密码等信息

处理原则：

- AI 输出中提高 `riskLevel`。
- 后端根据关键词兜底补充风险判断。
- AI 输出清晰写入高风险提醒。
- 前端置顶展示高风险提醒。
- 前端展示“先找家人确认”“联系官方渠道”两个提醒动作按钮。
- 不跳转到真实银行、医保、政务服务。
- 不触发电话、支付、登录等真实操作。

## 11. 部署方式

第一版 MVP 的部署目标是：简单可访问、方便演示、容易维护。

建议采用“前端静态页 + 一个轻量后端服务”的简化方案：

- 前端部署为静态 Web 页面。
- 后端部署为轻量 API 服务。
- AI 密钥只放在后端，不放在前端。

部署原则：

- 优先保证主流程稳定可体验。
- 优先保证演示可用，不追求复杂高可用架构。
- 优先保证配置简单、启动直接。
- 优先保证后端能安全保存 AI 调用密钥。

## 12. 不要做什么

- 不把 MVP 做成复杂业务平台。
- 不设计账户系统和用户中心。
- 不设计历史记录、收藏、分享等非核心功能。
- 不设计 OCR、语音识别、多模态上传。
- 不接入真实医保、银行、政务业务接口。
- 不把提醒动作按钮做成真实服务跳转。
- 不保存敏感个人信息。
- 不让前端直接暴露 AI 密钥。
