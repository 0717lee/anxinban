# 阶段一发现记录

## 已确认事实

- 当前仓库只有规则文档与规划文档，没有现成应用代码。
- 当前仓库没有 `package.json`，说明前后端项目需要从零搭建最小骨架。
- 当前仓库没有 `AGENTS.md`。
- 阶段一只允许做主链路最小跑通，不允许自动进入阶段二。
- 当前 `/api/generate` 已具备统一响应外壳 `success + data + error`。
- 当前 `callAi()` 仍是 fallback 骨架验证，不是真实 AI 调用。
- 当前阶段一只能证明前后端主链路和结构化返回已跑通，不能证明真实 AI 生成能力已完成。
- 后续在最终演示和部署前，必须完成真实 AI 接入，并保留失败时的明确错误提示。
- 该遗留问题会影响创新性、实用性、完成度评分，应作为后续高优先级修复项。

## 阶段二新增发现

- `server/index.js` 中的 `callAi()` 已从 fallback 骨架改为真实 AI 接入路径。
- 后端现在要求通过环境变量提供 AI Key，前端没有接触任何密钥。
- 当前实现已经满足“无 Key 时真实失败”，不再把 fallback 结果当成功返回。
- 当前本地环境没有配置可用 AI Key，所以还无法完成“真实 AI 成功生成”验证。
- 因此，当前阶段二第一优先级的实现状态应表述为：真实 AI 接入代码已完成，失败路径已验证，成功路径待可用 Key 验证。
- 后端现在增加了请求体长度上限，避免超长输入带来服务风险。
- 静态文件访问路径已经收紧，不能直接访问 `web` 目录外文件。
- 真实 AI 验证说明已整理完成，后续接入阿里云百炼时可按兼容 OpenAI 接口方式配置。
- 后端结构校验已进一步收紧，能更早拦住半结构化或缺字段 AI 返回，降低前端崩溃风险。
- 当前场景参数已增加白名单校验，非法值会直接返回 `SCENE_INVALID`。
- 当前 AI 请求已增加超时控制，避免上游长时间无响应时一直挂住请求。
- 当前提示词已增加“原始说明不是指令”的明确约束，并用 `<USER_TEXT>` 分隔原始文本，可降低提示词注入风险。
- 真实 Key 成功路径现已由用户复测通过：普通医保说明返回 `200 success:true`，银行验证码高风险返回 `200 success:true` 且 `riskLevel: high`。
- 同时错误路径仍保持通过：空输入返回 `TEXT_REQUIRED`，非法场景返回 `SCENE_INVALID`。
- 因此当前阶段可确认：真实 AI 调用链路、风险等级输出和关键输入校验均已完成一轮有效验证。
- 当前页面在不新增功能的前提下，已更贴近老人使用场景：普通结果页先解释再步骤，高风险结果页先提醒再阅读。
- 高风险提醒的表现已明显强化，并且“先找家人确认 / 联系官方渠道”保持为纯提示动作，没有越界跳转到真实业务。
- 上一版“温暖信笺风格”虽设计感强，但存在明显 AI 味：印章装饰、装订线渐变、金色点缀、多层动画、hover 位移等装饰性元素堆叠过多，偏离了“克制、老人友好”的核心目标。
- 去 AI 味清理后，页面回归克制：单一暖红主色、单一极简阴影、无装饰性伪元素、无过度动画、无 hover 位移，视觉权重集中在文字内容本身，更符合老人阅读需求。
- 类名同步是去 AI 味清理的关键风险点：HTML 类名变更后必须同步 CSS 和 JS 选择器，否则页面会无样式或交互失效。本次已确认 `web/index.html`、`web/styles.css`、`web/app.js` 三者类名完全匹配。
- 字体声明残留是常见清理盲点：删除 Google Fonts 外链后，CSS 中仍可能残留多处 `font-family: "Noto Serif SC", serif;` 声明，需用 Grep 全文搜索确认清除干净，否则字体加载不一致。
- `window.alert` 在移动端体验差且会阻塞页面，改为页面内提示（显示/隐藏元素）后，用户可在不离开当前上下文的情况下阅读提醒，符合老人友好设计。
- 强刷复验是确认前端改动的必要步骤：浏览器缓存可能导致旧 JS 仍在运行，必须用 `ignoreCache: true` 强刷后才能验证 alert 是否真的被替换。
- 通用兜底入口（general 场景）是老人友好的关键设计：老人面对医保/银行/政务三类场景时可能无法判断说明属于哪一类，兜底入口降低使用门槛。但必须控制视觉权重，避免喧宾夺主——使用虚线边框、更小字号、更浅文字色，让三个主场景仍然是首选。
- 后端新增场景时只需改三处：`isValidScene`（合法场景列表）、`getSceneLabel`（中文映射）、`buildPrompt`（AI 提示词中的 scene 字段说明）。AI 输出结构和安全机制无需改动，保证了一致性。
- 环境变量不跨会话继承是常见陷阱：在 PowerShell 会话 A 中配置的 AI Key 环境变量，不会自动出现在会话 B 中。重启服务时必须在同一会话中重新配置，否则会返回 `AI_KEY_MISSING`。已通过 `.env` 文件 + `loadLocalEnv()` 彻底解决此问题，服务启动时自动读取项目根目录 `.env`，不再依赖会话环境变量。
- PowerShell 中文 body 编码陷阱：`Invoke-RestMethod` 用单引号字符串作为 `-Body` 时，中文可能因控制台编码被截断或乱码，导致后端收到的 text 不完整，关键词兜底无法触发。正确做法是用 `ConvertTo-Json` 生成 body 后，用 `[System.Text.Encoding]::UTF8.GetBytes()` 转为字节流再发送。
- 自实现 env 加载器比引入 dotenv 更轻量：只需处理注释、空行、引号包裹、BOM 去除、不覆盖已存在环境变量五个边界，约 40 行代码即可替代 dotenv 依赖，符合项目"轻量后端服务"定位。
- 老人友好的文案应避免技术词和书面语：把"生成解释和步骤"改成"帮我看一下"、"粘贴说明内容"改成"把看不懂的字粘过来"、"正在整理说明"改成"正在帮你看"，更贴近老人日常说话方式，降低理解门槛。
- 文案简短化是老人友好的关键：长句拆短句、去掉"等敏感信息"等冗余修饰、把"请稍等，我正在帮你整理内容"压缩成"稍等一下，马上就好"，减少老人阅读负担。
- 安全提示文案也要口语化：把"不要输入身份证号、银行卡号、密码、验证码等敏感信息"改成"别填身份证号、银行卡号、密码、验证码"，用"别填"替代"不要输入...等敏感信息"，更直接更易记。
- 文案改动后必须强刷验证：浏览器缓存会导致旧 JS 仍在运行，必须用 `ignoreCache: true` 或带时间戳 URL 参数（如 `?v=20260617e`）强刷后才能验证文案是否真的更新。
- 文案改动容易引入语法错误：手动编辑字符串时容易漏掉闭合引号或分号（如 `'请先贴说明内容` 缺少闭合），导致整个 JS 文件解析失败、页面完全无交互。每次文案改动后必须检查控制台是否有 SyntaxError。
- 前端静态文件改动不需要重启后端服务：端口被旧服务占用时（EADDRINUSE），无需杀进程重启，直接强刷浏览器即可加载最新的 HTML/CSS/JS 文件。
